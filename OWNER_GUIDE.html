JARVIS Owner Guide v3.0.0-rc3 — Local-First AI Workspace

A local-first AI workspace that can plan, execute, observe, and learn from real-world tasks using interchangeable capability providers.

Architecture Overview

JARVIS is built on a three-layer abstraction: Goal → Capability → Provider. The planner never knows about specific tools, models, or APIs — it only routes goals to capabilities, and the capability graph resolves providers at runtime.

The planner requests coding. The capability graph checks permissions. The provider router selects the best provider (Claude if online, Qwen if offline). Execution happens. The result reaches the activity graph and knowledge store for learning.

4.2 Scheduler

The scheduler runs as a background asyncio task. It picks the highest-scored ready activity and delegates execution to the resume engine → planner state machine → agent graph → workflow engine.

Priority factorWeightPurpose
Priority (0–5)0,20,40,60,80,100Manual priority
Urgency+30Ready/running gets boost
Retry+50Failed attempts get retry priority
Waiting time+2/minStale activities rise
User requested+80User goals over subgoals

4.3 Automation Examples

"Research AI news every evening"Creates recurring research task. Planner decomposes: research → extract → synthesize. Scheduler triggers daily.
"Build coffee shop app and email the APK"Parallel feature decomposition (loyalty, payments, admin, customer, analytics) → build → test → email.
"Watch a folder for new PDFs"Desktop + filesystem monitors directory. On new file → research extraction → summarize → store.

5.2 Provider Routing

When the planner requests a capability, the ProviderRouter in core/providers/router.py scores every registered provider using context-aware calibration. The highest-scoring capable provider is selected.

Example trace

Goal: "Build a portfolio website"
Capability needed: coding
Providers scored:
  • ollama/qwen2.5:7b  — score 0.82 (local, available, good for code)
  • openai/gpt-4       — score 0.91 (cloud, higher quality, API key present)
Selected: openai/gpt-4
Reasoning: Higher quality score, API key configured, no budget constraints

9. Glossary

TermDefinition
CapabilityAn abstract skill JARVIS can perform (coding, research, browser, desktop). Defined in the Capability Graph.
ProviderA concrete implementation of one or more capabilities (Claude, Playwright, GitHub API). Each has a manifest + lifecycle.
PlannerDecomposes goals into sub-goals and enforces execution order. Template-driven, state-machine-controlled.
RouterSelects the best provider for a given capability using context-aware calibration scoring.
Negotiation EngineResolves permission conflicts and consent requirements between user and providers.
Activity GraphPersistent DAG storing every goal, sub-goal, agent, tool call, and artifact lineage.
Workflow EngineDurable step execution with retry, compensation, crash recovery, and context persistence.
Artifact StoreFilesystem-backed, checksummed artifact registry. Supports cross-system references via artifact: prefix.
FSMFinite State Machine. Three FSMs exist: BrowserFSM (9 states), LongHorizonFSM (10 states), ExtractionFSM (10 states).
CalibrationPer-context outcome tracking that tunes provider selection weights over time.
KnobA tuneable system parameter controlled via Phase 10 adaptive behavior loop. 11 defined knobs.
SDKProvider SDK for third-party providers. Enables installable providers without modifying JARVIS core.
